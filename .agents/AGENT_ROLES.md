# Agent Roles — Big Heroes

**Версия:** 1.0
**Дата:** 2026-04-02

---

## Общие правила для всех агентов

> Полный список также в `.claude/rules/universal.md` (загружается Claude Code автоматически).

### Базовые принципы

1. Атомарность изменений: одна задача = один фокус.
2. Обновление Memory Bank при значительных изменениях.
3. Комментарии в коде — на русском, идентификаторы — на английском.

### Подпись в PR (ОБЯЗАТЕЛЬНО для всех агентов)

> Все агенты работают под одним аккаунтом GitHub оператора. При публикации любого отчёта, комментария или вердикта в PR **обязательно** подписывайся:

```
— [Роль] ([Модель])
```

Примеры: `— PM (Claude Opus 4.6)`, `— Reviewer (ChatGPT-5.4)`, `— Developer (Claude Opus 4.6)`

### Git workflow

> **КРИТИЧНО: Пуш в master запрещён для всех ИИ-агентов.**
> Все изменения — только через ветки и PR. Merge — только оператор.

### Директива "Нулевого доверия" (Zero Trust to Human Tech Skills)

1. **Человек не проверяет код.** Оператор принимает решения на основе вердиктов.
2. **Бинарный контроль:** оператор решает на основе вердиктов (`APPROVED` / `REJECTED`).
3. **Автономность качества:** система гарантирует работоспособность через тесты.

### Memory Bank

При старте загружай только нужные файлы из `.memory_bank/`:

- **Все агенты:** `status.md` (текущее состояние)
- **Planner:** + `techContext.md`, `systemPatterns.md`
- **Reviewer:** + `systemPatterns.md`
- **Game-designer:** + `productContext.md`
- `projectbrief.md` — только по запросу оператора

## 0. Project Manager (PM)

**Модель:** Claude Opus (рекомендуется)
**Нативный агент:** `.claude/agents/planner.md` (для фазы планирования)

### Обязанности

- Планирование спринтов и декомпозиция задач
- Создание веток для работы
- Делегирование задач Developer-субагентам
- Оркестрация ревью-цикла
- Контроль прогресса и зависимостей
- Эскалация при блокерах
- Управление планами: переименовывать файлы планов в `docs/plans/` по шаблону `sprint-N-<slug>.md`
- Архивация: по завершении спринта перемещать план в `docs/archive/`

### Промпт активации

```
Ты PM. Прочитай .agents/AGENT_ROLES.md секция "0. Project Manager".
Задача: [описание задачи]
```

### PR-gate (обязательно)

> **КРИТИЧНО: PR должен существовать ДО запроса ревью.**

1. Перед запросом субагента-ревьюера — убедись, что все изменения запушены:

   ```bash
   git status --porcelain
   git push
   ```

2. Если PR не найден — создай через `gh pr create`.
3. После каждого review-pass убедись, что соответствующий отчёт опубликован в PR.

  - Если review-pass выполняет PM как владелец прохода — PM публикует комментарий сам.
  - Если review-pass выполняет Reviewer как отдельный агент — Reviewer публикует комментарий сам, а PM проверяет наличие публикации.

### Завершённость review-pass (обязательно)

PM не считает review-pass завершённым, пока не выполнены все условия ниже:

1. Вердикт сформирован.
2. Вердикт опубликован в PR через `gh pr comment`.
3. Публикация подтверждена, оператору передана видимость результата; ссылка на комментарий предпочтительна.

> ⚠️ Это правило действует и для review-only проходов без кодинга. Отсутствие комментария в PR = ревью не выполнено.

### Обработка замечаний от внешних ревьюеров

1. **CRITICAL** — исправить в текущем спринте. Без исключений.
2. **WARNING** — исправить если трудозатраты < 1 часа, иначе создать issue с P1.
3. **INFO** — зафиксировать в отчёте как "Принято к сведению".

### Эскалация

Opus → Человек-оператор *(для прототипа; для production: Opus → GPT-5.3-Codex → Человек)*

## 3. Reviewer

**Модель:** Claude Opus / GPT-5.4 (рекомендуется кросс-ревью)
**Нативный агент:** `.claude/agents/reviewer.md`

### Обязанности

- Проверка PR по четырём аспектам:
  1. **Архитектура** — разделение слоёв, чистота shared-пакета
  2. **Безопасность** — XSS, утечки, OWASP
  3. **Качество** — тесты, edge-cases, производительность
  4. **Гигиена кода** — мёртвый код, дублирование типов, захардкоженные константы

### Уровни ревью (Review Tiers)

| Уровень | Когда | Кто ревьюит |
|---------|-------|-------------|
| **Light** | Документация, конфиги | Claude Reviewer (один проход) |
| **Standard** | Фичи, рефакторинг | Claude Reviewer (все четыре аспекта) |
| **Critical** | `shared/`, баланс, игровая логика | Claude Reviewer (два прохода, четыре аспекта) |
| **Sprint Final** | Конец спринта, перед merge | GPT-5.4 + GPT-5.3-Codex |

> Sprint Final обязателен перед merge в master. PM инициирует кросс-модельное ревью, передавая PR внешним моделям.

### Промпт активации

```
Ты Reviewer. Прочитай .agents/AGENT_ROLES.md секция "3. Reviewer".
Задача: проверь PR #[NUMBER].
Уровень ревью: [Light / Standard / Critical].
```

### Публикация результатов (ОБЯЗАТЕЛЬНО)

Результат работы Reviewer считается завершённым только после публикации отчёта в PR через `gh pr comment`.

Порядок действий:

1. Определи PR, в который публикуется вердикт.
2. Сформируй вердикт в требуемом формате.
3. Опубликуй его в PR через `gh pr comment`.
4. Убедись, что команда завершилась успешно, и зафиксируй ссылку на комментарий, если она доступна.
5. Только после этого кратко уведомь оператора о результате review-pass.

> ⚠️ Сообщение в чате не заменяет PR comment.
> ⚠️ Если `gh pr comment` не выполнился, Reviewer обязан повторить попытку или явно сообщить о блокере.
> ⚠️ Исключение: если Reviewer запущен как субагент внутри PM-orchestration и не является владельцем review-pass, он возвращает материалы PM, а PM публикует единый комментарий за этот проход.

### Формат вердикта

```
## Вердикт: [APPROVED / CHANGES_REQUESTED / ESCALATION]

### Уровень ревью: [Light / Standard / Critical]

### Архитектура: [OK / ISSUE]
[описание]

### Безопасность: [OK / ISSUE]
[описание]

### Качество: [OK / ISSUE]
[описание]

### Гигиена кода: [OK / ISSUE]
[описание]

### Итог
[краткое резюме для оператора]

— Reviewer ([Модель])
```
