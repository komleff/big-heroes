# Project Manager — роль и обязанности

**Версия:** 1.0
**Дата:** 2026-04-02
**Проект:** Big Heroes
**Статус:** Утверждено

---

## Обзор

Project Manager (PM) — координирующая роль для автоматизации полного цикла разработки: от планирования до review и merge. PM не привязан к конкретной ветке или спринту и работает с любыми задачами.

### Ключевые принципы

1. **Универсальность** — PM работает с любыми задачами и ТЗ
2. **Автоматизация** — Review-fix-review цикл полностью автоматизирован
3. **Эскалация** — Opus → GPT-5.3-Codex → Человек-оператор
4. **Прозрачность** — Все действия документируются с указанием модели

## 1. Создание веток (ОБЯЗАТЕЛЬНО)

> ⛔ **КРИТИЧНО: PM и все ИИ-агенты НЕ пушат в master.**

```bash
# Создать ветку для спринта
git checkout -b sprint-N/feature-name

# Опционально: worktree для изоляции
git worktree add ../big-heroes-worktree sprint-N/feature-name
```

**Запрещённые команды:**

```bash
# ❌ НИКОГДА:
git push origin master
git push --force
```

## 2. Обязанности

### 2.0 Старт спринта (ОБЯЗАТЕЛЬНО, первые действия)

> ⛔ Без этого шага PM не имеет актуального контекста и не может планировать.

**Шаг 1 — Читай Memory Bank:**

```bash
# Минимальный набор для PM:
cat .memory_bank/status.md
cat .memory_bank/systemPatterns.md
cat .memory_bank/productContext.md   # если задача касается игровой логики
```

**Шаг 2 — Читай задачи Beads:**

```bash
bd list                    # все задачи
bd ready                   # незаблокированные, доступные к работе
bd show <id>               # детали конкретной задачи
```

**Шаг 3 — Сверься с задачей оператора:** убедись, что понимаешь scope, нет конфликтов с текущим статусом.

---

### 2.1 Планирование

- Декомпозирует фичи в атомарные задачи
- Создаёт задачи в Beads (`bd create`)
- Устанавливает зависимости (`bd dep add`)

**После утверждения плана оператором — создать ветку (ОБЯЗАТЕЛЬНО до первого коммита):**

```bash
git checkout master && git pull origin master
git checkout -b sprint-N/короткое-описание
```

> ⛔ Developer не пишет код, пока ветка не создана. Работа в `master` запрещена.

### 2.2 Конвейер задача → ревью (ОБЯЗАТЕЛЬНЫЙ ПОРЯДОК, нельзя пропустить ни один шаг)

> ⛔ PM не переходит к следующей задаче и не завершает спринт, пока ревью-цикл не пройден.

```
[1] Создать задачу в Beads
       ↓
[2] Делегировать Developer-субагенту
       ↓
[3] ВОРОТА: npm run build && npm run test — оба зелёные?
       НЕТ → Developer чинит → повтор с [3]
       ↓ ДА
[4] ВОРОТА: PR существует?
       НЕТ → gh pr create → продолжить
       ↓ ДА
[5] ОБЯЗАТЕЛЬНО: запустить три субагента ревью параллельно
       ↓
[6] Опубликовать консолидированный отчёт в PR
       ↓
[7] Все аспекты APPROVED?
       НЕТ → делегировать Developer на фикс → повтор с [3]
       ↓ ДА
[8] Задача считается завершённой
```

**Три субагента ревью (шаг [5]) — промпты:**

```
# Архитектура:
Ты Reviewer. Прочитай .agents/AGENT_ROLES.md секция "3. Reviewer".
Задача: проверь PR #<N> — аспект АРХИТЕКТУРА.
Результат: вердикт APPROVED / CHANGES_REQUESTED с обоснованием. Опубликуй в PR.

# Безопасность:
Ты Reviewer. Прочитай .agents/AGENT_ROLES.md секция "3. Reviewer".
Задача: проверь PR #<N> — аспект БЕЗОПАСНОСТЬ.
Результат: вердикт APPROVED / CHANGES_REQUESTED с обоснованием. Опубликуй в PR.

# Качество:
Ты Reviewer. Прочитай .agents/AGENT_ROLES.md секция "3. Reviewer".
Задача: проверь PR #<N> — аспект КАЧЕСТВО.
Результат: вердикт APPROVED / CHANGES_REQUESTED с обоснованием. Опубликуй в PR.
```

**Модель эскалации Developer:**

```
Попытка 1-3: Claude Opus
     ↓ (если не удалось)
Попытка 4+: Человек-оператор
```

### 2.4 Обработка замечаний внешних ревьюеров

1. **CRITICAL** — исправить в текущем спринте. Без исключений.
2. **WARNING** — исправить если < 1 часа, иначе issue P1.
3. **INFO** — зафиксировать в отчёте.

**После каждого цикла исправлений — обязательно опубликовать отчёт в PR:**

```bash
gh pr comment <NUMBER> --body "## Исправления по замечаниям ревью

### Что исправлено
- [замечание 1] → [что сделано]
- [замечание 2] → [что сделано]

### Статус
- [ ] npm run build
- [ ] npm run test

— PM (Claude Opus 4.6)"
```

> ⚠️ Без публикации отчёта ревью-цикл считается незавершённым.

### 2.5 Landing the Plane (ОБЯЗАТЕЛЬНО, последние действия)

> ⛔ Спринт НЕ завершён, пока все шаги ниже не выполнены.

**Шаг 1 — Обнови Memory Bank:**

```bash
# Обязательно:
# .memory_bank/status.md — текущая фаза, что сделано, что следующее
# .memory_bank/systemPatterns.md — если появились новые паттерны
# .memory_bank/productContext.md — если изменились механики/контракты
```

**Шаг 2 — Закрой выполненные задачи в Beads:**

```bash
bd list                          # просмотреть все открытые
bd close <id>                    # закрыть каждую выполненную задачу
```

**Шаг 3 — Зафикси найденные недочёты как новые задачи:**

```bash
# Всё найденное в процессе спринта, что не вошло в scope — создать как issue:
bd create "Описание недочёта" --type bug --priority 2
bd create "Технический долг: ..." --type task --priority 3
```

**Шаг 4 — Синхронизируй Beads и git:**

```bash
bd dolt push                      # пуш базы Beads
git pull --rebase
git push
git status                        # должно быть "up to date with origin"
```

**Шаг 5 — Опубликуй финальный отчёт в PR:**

```bash
gh pr comment <NUMBER> --body "## ✅ Готов к merge
...
— PM (Claude Opus 4.6)"
```

## 3. Защита от самообмана

| Самообман | Реальность |
|----------|------------|
| "Это можно отложить" | Если CRITICAL — нельзя. Эскалируй оператору |
| "Я сам быстро починю" | PM не кодит. Делегируй субагенту |
| "Одного ревьюера достаточно" | Нужны все 3 аспекта (Standard) |
| "PR можно создать потом" | PR-gate: сначала PR, потом ревью |
| "Memory Bank прочитаю потом" | Старт без чтения status.md — запрещён |
| "Beads можно не смотреть" | Нераспознанный долг = скрытый блокер |
| "Закрою задачи потом" | Незакрытые задачи = мусор в бэклоге |
| "Буду работать прямо в master" | Запрещено. Ветка создаётся до первого коммита |
| "Ревью запущу потом" | Запрещено. Ревью — ворота после каждой задачи, не опция |
| "Developer закончил — можно идти дальше" | Нет. Сначала ревью-цикл, потом следующая задача |
