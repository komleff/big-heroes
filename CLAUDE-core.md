# CLAUDE-core.md — Базовые правила (загружается всегда)

---

## 1. Режим работы по умолчанию: диагностика

1. Найти требование (`REQ-*`, `ACC-*`, раздел документа), которому противоречит поведение.
2. Сформулировать проблему и её влияние на релиз (блокирует / не блокирует).
3. Предложить минимальное исправление.
4. Описать проверки, подтверждающие исправление.
5. Не менять код без явной команды «внести правку» или «сделать PR/патч».
6. При расхождении требований между документами — сообщить оператору с указанием конфликтующих мест.

**Запрещено:**

- «Улучшать» стиль, форматирование, архитектуру без запроса.
- Массово переименовывать, переносить файлы, менять публичные контракты.
- Добавлять зависимости без запроса.
- Исправлять «заодно» несвязанные места.

---

## 2. Правила составления ТЗ

- Язык: только русский, без жаргона и англицизмов.
- Технические названия (переменные, классы, файлы, endpoint) — на английском.
- Стиль: кратко, по делу, без вводных слов.
- Содержание: только логика, требования и архитектура словами.
- Примеры кода и конкретные реализации запрещены.
- Неопределённости выносить в раздел «Открытые вопросы».

---

## 3. Правила правок (если правка разрешена)

1. Делать только минимально необходимое для выполнения конкретного `REQ` или устранения дефекта.
2. Не менять API без обновления архитектурного документа.
3. Добавлять защитные проверки и логирование только вокруг места дефекта.
4. В описании результата перечислить, какие `ACC-*` выполняются.
5. Комментарии в коде — только на русском. Объясняй «почему», а не «что». Идентификаторы — на английском.

---

## 4. Приоритеты дефектов

| Приоритет | Описание |
|-----------|----------|
| P0 | Потеря данных, критические падения, недоступность ключевых систем |
| P1 | Деградация сервисов, ошибки конфигов, массовые сбои |
| P2 | UI-дефекты, вторичные проблемы, локальные оптимизации |
| P3 | Косметика, улучшения UX, технический долг без влияния на функциональность |

---

## 5. Именование требований

- `REQ-<DOC>-NNN` — требование (NNN: 001–999).
- `ACC-<DOC>-NNN` — критерий приёмки.
- `DOC` — короткий идентификатор: `ARCH`, `SPEC`, `GDD`, `API` и т.п.

---

## 6. Формат ответа при диагностике

1. **Заголовок:** кратко «что сломано».
2. **Источник правды:** ссылка на документ/раздел/REQ.
3. **Локализация:** модуль/файл.
4. **Гипотеза причины.**
5. **Минимальное исправление** (шаги 1…N, без кода).
6. **Риски.**
7. **Проверки** (какие тесты/скрипты прогнать).
8. **Что НЕ трогать.**
